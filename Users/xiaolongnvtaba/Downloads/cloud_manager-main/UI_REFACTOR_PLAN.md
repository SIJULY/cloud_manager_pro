# Cloud Manager UI 重构设计思路

## 目标
在不改动任何业务功能、接口、交互流程、字段和按钮语义的前提下，将 `cloud_manager-main` 的整体 UI 重构为接近 `x-fusion-panel-pro` 的深色科技控制台风格。

## 核心原则
1. 仅改 UI，不改功能逻辑。
2. 保持现有 Flask/Jinja 模板结构与 JS 事件绑定兼容。
3. 优先改布局、配色、卡片、表格、弹窗、登录页、导航。
4. 不修改接口路径、DOM 关键 id、表单 name、按钮 id。
5. 尽量通过：
   - `templates/layout.html`
   - `templates/login.html`
   - `templates/aws.html`
   - `templates/azure.html`
   - `templates/oci.html`
   - 新增统一主题 CSS
   完成视觉升级。

## 新风格方向
- 深色主背景：`#0f172a`
- 二级面板：`#111827` / `#1e293b`
- 边框：`#334155`
- 主品牌色：蓝紫系 `#6366f1` / `#8b5cf6`
- 状态色：成功绿 / 危险红 / 警告橙 / 信息蓝
- 关键词：科技感、控制中心、监控台、深色玻璃卡片、发光边框、低饱和霓虹强调

## 布局重构方案
### 1. 全局布局
- 顶部传统 navbar 改为更强品牌感的深色 header
- 主内容区改为 dashboard 式容器
- 页面标题区统一成 Page Hero/Header
- 卡片统一圆角、边框、阴影、悬浮反馈

### 2. 登录页
- 改为深色全屏背景
- 中央科技感登录卡片
- MFA 区块、错误提示、按钮统一主题
- 保留原有字段与提交逻辑

### 3. 主页面
每个云平台页面统一为四层结构：
1. 页面头部（标题 + 平台标识 + 状态）
2. 顶部概览/快捷操作区
3. 主工作区（账户、区域、操作）
4. 资源列表 + 日志区

### 4. 卡片体系
统一卡片为：
- 面板卡（主内容）
- 功能卡（操作入口）
- 状态卡（数量/状态）
- 日志卡（终端风格）

### 5. 表格体系
- 深色表头
- hover 高亮
- selected 行蓝紫高亮
- sticky header 保留
- 状态 badge / dot 统一

### 6. Modal 体系
- 统一深色弹窗
- 顶部标题栏加强层级
- 表单输入统一暗色风格
- 危险操作统一 danger 风格

## 页面级改造重点
### AWS
- 双栏头部区：账户管理 / 账户列表
- 操作区重组为科技卡片
- 实例表格与日志统一深色面板

### Azure
- 跟 AWS 保持相同骨架
- 强化创建 VM 区和实例列表层级

### OCI
- 作为最复杂页面，优先做视觉体系统一
- 保留所有 modal 和复杂操作，但统一为深色控制台风格
- 任务列表、网络设置、TG/API、Cloudflare 等弹窗统一皮肤

## 实施顺序
1. 建立统一主题 CSS（全局变量、按钮、卡片、表格、modal、表单）
2. 重构 `layout.html`
3. 重构 `login.html`
4. 重构 `aws.html`
5. 重构 `azure.html`
6. 重构 `oci.html`
7. 逐页检查 JS 兼容性
8. 统一细节与响应式

## 风险点
1. 不能改动 JS 依赖的元素 id / 结构过深层级。
2. Bootstrap 默认组件覆盖需谨慎，避免影响 modal、dropdown、table JS 行为。
3. OCI 页面复杂度高，需分阶段改，避免一次性改动过大。
4. 如果后续要完全贴近第一个项目，可能需要增加较多自定义 CSS，但仍不改功能。

## 建议工作流
- 第一阶段：全局主题与骨架
- 第二阶段：登录页 + layout
- 第三阶段：AWS / Azure 页面统一重构
- 第四阶段：OCI 页面深度重构
- 第五阶段：细节修正与验收
